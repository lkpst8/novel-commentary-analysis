# Large Novel Scaling Guide

## Use This Reference

Use this file when the novel is large enough that the main risk is no longer basic understanding, but omission. This usually happens with multi-megabyte or tens-of-megabytes source files, long serialized web novels, ensemble epics, or stories with many subplots and recurring character waves.

## Input Strategy

Do not treat a huge novel as one document to summarize once.

Use a layered workflow:

1. `Packet layer`
   Read small packets in order and extract concrete events, character updates, world notes, subplot movement, and unresolved threads.

2. `Phase layer`
   Group several packets into one phase and summarize what changed at the story level.

3. `Global ledger layer`
   Maintain a running record for:
   - characters
   - relationships
   - factions/institutions
   - world rules
   - main plot progression
   - subplot states

4. `Final HTML layer`
   Convert the ledger into a readable page.

5. `Compression layer`
   Build reusable medium and short outline artifacts so later output does not need to reread the raw text.

## Packet Discipline

For each packet, extract:

- key events
- newly relevant characters
- changes in known characters
- changes in world rules or institutions
- subplot openings or escalations
- unresolved threads

Never let a packet pass without writing what it contributed.

Once packet notes exist, treat them as the preferred source for later synthesis.

## Phase Discipline

Each phase summary should answer:

- what changed in the main line
- which subplots moved
- which characters entered, changed, or exited importance
- whether the background/world understanding changed
- what the next phase inherits

## Anti-Omission Ledger

Before final HTML generation, confirm:

- every packet is represented in the ledger
- every major character appears in at least one character section
- every important subplot has start, development, and closure status when source permits
- late-phase revelations are reflected in earlier interpretive claims where necessary
- the ending section reflects the full book rather than only the early or middle arcs

## Long HTML Strategy

For huge novels, the HTML should not be a single flat wall of text.

Use layers:

### Layer 1: Fast understanding

Include:

- one-sentence positioning
- story snapshot
- background/world summary
- main characters at a glance
- main plot in compact stages

### Layer 2: Structured depth

Include:

- phase-by-phase breakdown
- named subplot section
- turning-point section
- ending and aftermath
- theme and emotional core

### Layer 3: Overflow control

Use when the book is especially large:

- supporting cast appendix
- faction appendix
- detailed timeline appendix
- optional `details/summary` collapsible blocks

This keeps the page readable without deleting important material.

## Compression Rules

When the page becomes too large:

- shorten sentences before deleting events
- merge repetitive minor beats into one summary line
- move secondary detail into appendices
- keep the causal chain intact even when compressing
- never compress away a major turn, a major character shift, or a subplot resolution

## Red Flags

Your summary is probably too shallow if:

- the protagonist is clear but the world is vague
- the first half is detailed but the later half is rushed
- support characters disappear from the explanation after introduction
- subplots are named but not connected back to the main line
- the ending is only described in one vague paragraph

If any of those happen, go back to the phase summaries and coverage ledger before finalizing.

## Token-Saving Rule

For huge novels, every unnecessary reread of raw source is expensive.

Preferred order of reuse:

1. manifest and chapter index
2. packet notes
3. phase notes and phase summaries
4. ledgers
5. compression passes
6. final HTML or final short outline

Only fall back to raw packets when a specific uncertainty still matters.
